- always read the .md file in every folder you work in. if there is no .md file in this folder create it if it is in the backend folder for example then it should be called backend_developer_guide.md, if you are creating the .md file make sure it is written in a way that is geared towards developers, it should explain all the files and subfolders in that folder in a way that a developer that has never seen the code will know how to work with the code. these files should never be over 500 lines long.
- when changing any file or doing any edits make sure to update the .md file in that folder and in the ancestor folders as well when needed. these .md files should never be longer than 500 lines long. the point of these .md files are to allow a new developer to understand everything without needing to read the code itself. they are not to document changes. only changes that require additional information for a developer that has never seen the code should be added into the .md files.
- IMPORTANT: Follow DRY (Don't Repeat Yourself) in documentation. Detailed feature documentation should live in ONE place (the most specific/relevant location). Higher-level MD files should provide navigation links to detailed docs, NOT duplicate the content. Example: detailed implementation of a feature in map_tab.py belongs in gui/tabs/tabs_developer_guide.md; parent folders should link to it rather than repeating the same information.
- no code file generated or edited should exceed 1000 lines of code ever. if while editing or creating a file, the file exceeds 1000 lines this file must be split up into multiple files.
- no folder should contain more than 10 code files. if we need to split a file which will cause the folder to go over 10 code files we must consider how to split up the files into sub directories.
- whenever creating a new file, think very carefully which folder this file should be put in, if needed make a new folder so that there is clear separation between folders.
- each folder should only have 1 .md file. never create summary files or visualization .md files.
- `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md` are exempt from the one-Markdown-file-per-folder rule and should be treated as instruction files rather than developer guides.
- never mention legacy functionality or recent changes in the MD files they are only needed to get engineers into what is happening right now in the code.
- always ask if a commit message is good before commiting.
