# Game Package Guide

## Purpose

`com.example.cw.game` contains the full prototype gameplay loop: screen flow, campaign state, match simulation, canvas rendering, and world helpers.

## Files

- `CampaignPersistence.kt`: SharedPreferences-backed campaign save and load helpers plus compact serialization for completed levels, per-level star records, bonus star credits, spent-star totals, the persistent campaign upgrade levels for mission income, player-node refill rate, player fleet speed, and the special-ability loadout state. Saves write an explicit schema version so unsupported future formats can be rejected instead of being parsed as partial progress. Schema-specific migration treats `Cash Flow` as a v2 field, refill/speed upgrades as v3 fields, and the selected special ability plus per-ability upgrade levels as v4 fields; each newer field is explicitly zeroed or cleared for older saves instead of relying only on missing-key defaults. When adding any new persisted `CampaignState` field, add its schema gate in `decodeCampaignState` in the same change so older saves cannot read a newer key by accident.
- `GameApp.kt`: Top-level Compose entry for the game feature. Owns screen navigation, campaign state, active match state, packaged level loading, the frame-timed simulation loop, and persistence wiring for campaign progress. Campaign progress records each level's best star result, persists across app restarts, only adds the best-star improvement delta to the player's spendable star reserves, only mutates campaign progression on the `RUNNING -> PLAYER_WON` transition, and snapshots the current campaign refill-speed bonuses plus the equipped special-ability loadout into each new or restarted match. Android system back handling is centralized here as a pure state transition: from levels or upgrades it returns home, from the special-abilities screen it returns to upgrades, from a running mission it opens the pause menu, and from a paused or completed mission it leaves the mission for the level list. The `createMatch` and restart call sites carry the snapshot comment because that is where the design decision is expressed; the per-frame loop intentionally keys only on screen and match lifecycle state, and cash income alone reads the latest campaign state each frame.
- `GameScreens.kt`: Menu-only UI for the home screen, asset-backed level selection, the upgrade screen, and the dedicated special-abilities loadout screen. These screens surface ready-to-spend stars separately from lifetime earned stars, each level card derives a validated `LevelCardState` before reading the shared palette and status label helpers so impossible combinations such as "completed but locked" are rejected at the helper boundary, and the upgrade flow remains the only place that spends meta-progression currency while exposing `Cash Flow`, `Refill Rate`, `Fleet Speed`, and the per-ability upgrade cards for the special-ability loadout. Campaign upgrades are capped and the cards switch to a maxed state instead of allowing infinite purchases, while the abilities screen keeps only one ability equipped at a time but preserves each ability's stored upgrade level when the player switches loadouts.
- `GameInGameUi.kt`: In-match HUD, pause overlay, level-end overlay, the selected-base upgrade affordance, and small in-game UI formatting helpers. The top HUD keeps the player economy readout as a compact floating gold `$` amount at the left edge, a live mission timer on its own centered chip directly below the funds/pause row using the same `m:ss` formatter as the results overlay, an icon-only pause action with no surrounding chip background, and a second centered chip for the equipped special ability that shows whether the ability is ready, active, or recharging. Shared HUD card surfaces now come from `GameUiKit.kt` instead of inline literals, while reward text comes from Android plural resources. Node upgrades trigger a five-second gold ring-fill animation around the upgraded base; the trigger logic is derived from the selected player base's `capLevel` transition instead of being stored as a separate gameplay event, and the animation runner always clears its overlay state in `finally` so selection changes cannot leave a frozen ring behind.
- `GameCanvas.kt`: Custom canvas rendering for the starfield, obstacles, bases, fleet trails, and fleets. Base labels show current unit count in the center and render a slightly larger base-level badge in a circular badge that overlaps the node's bottom edge using a relative badge offset so the geometry stays readable without competing with the garrison count; the badge radius and stroke width scale from the rendered node radius, while the numeral size clamps using density-aware `sp` bounds converted to pixels inside the composable so the level number stays legible across device densities. When a base is near the bottom of the viewport, the badge clamps upward enough to remain visible. Affordable player bases that can still level up render a small gold upward upgrade arrow above the node, with the arrow path built once per draw call and outlined with the shared `NodeUpgradeIndicatorOutline` token instead of an inline hex literal. Selection feedback stays visual through the aura and stroke treatment, without a separate text tag above the node.
- `GameUiKit.kt`: Shared visual tokens and reusable Compose primitives such as the menu background, divider, action buttons, stat rows, the named surface colors reused by the level-card palette helpers, and the UI-layer color mapping for `SpecialAbilityType`. Keep new cross-screen color tokens here rather than scattering raw hex literals through individual composables or styling helpers. `NodeUpgradeIndicatorOutline` is intentionally a dark shadow/border for the gold upgrade arrow so the arrow still separates cleanly from the starfield without reading as a second bright accent.
- `GameLogic.kt`: Input handling, fleet launching, special-ability activation, base upgrades, frame stepping, AI decisions, ship production, combat resolution, arrival handling, capture-level downgrade handling, and selection cleanup when ownership changes. `stepMatch` treats pause as a hard no-op so elapsed time, economy, and ability timers cannot advance if a caller accidentally invokes it while paused, clears short-lived HUD hints after they expire, and empty-space taps now clear the current player selection without overwriting the active HUD message. Successful tap-driven launches and reinforcements are now intentionally silent so routine attacks do not spam the HUD banner; failure states such as insufficient ships still use the banner, and a valid launch clears any transient tap hint instead of leaving stale guidance visible. Campaign refill-rate upgrades multiply only player-owned node production, while campaign fleet-speed upgrades multiply only player-launched fleet velocity. Special abilities now run inside live matches: `Speed Burst` accelerates player fleets for five seconds, `Defense Field` reduces incoming enemy attack power against player-held nodes for five seconds, `Attack Surge` boosts player fleet damage for five seconds, and `Instant Refill` immediately fills the selected player node to its cap and then grants that node a five-second production surge. Every ability uses a 30 second recharge.
- `GameWorld.kt`: Match creation from authored level definitions, route building around obstacles, coordinate conversion, and shared world geometry helpers. New matches start with an empty HUD banner so authored helper text does not cover the playfield, snapshot the equipped campaign special ability plus its level into `MatchState`, and obstacle avoidance now builds a shortest route across perimeter waypoint rings around every obstacle instead of relying on a single detour point, which keeps each route segment outside the obstacle collision buffer. Generated waypoint nodes clamp to named side and vertical world margins before the graph search runs, and the helper intentionally falls back to the direct endpoint if no graph path can be built.
- `LevelCardStyle.kt`: Shared state, palette, and status helpers for campaign level cards. The file defines the explicit `LevelCardState` used by `GameScreens.kt`, keeps the locked-status copy null-safe when a prerequisite level id is unavailable, and exposes palette-owned status letter spacing so card rendering stays declarative while tests pin the contrast-oriented state mapping.
- `GameModel.kt`: Immutable state models and enums for campaign progress, match state, world bounds, bases, fleets, obstacles, ownership, base types, and special-ability loadouts. Campaign state stores per-level star records, derives lifetime earned stars from those records, tracks spent stars separately for the upgrade economy, caps campaign upgrades at `CAMPAIGN_MAX_UPGRADE_LEVEL`, stores the currently equipped special ability plus per-ability upgrade levels, and exposes upgrade-derived multipliers for mission cash, player-node refill rate, and player fleet speed. Match state tracks elapsed time plus the stars and reserve gain earned on the current result, carries the per-match player production and speed multipliers that were active when the mission started, and now also holds the equipped special ability, its snapshot level, the active/cooldown timers, and the temporary target base used by instant-refill surges.

## Subfolders

- `levels/`: Shared authored level schema, JSON parsing, repository loading, and campaign unlock helpers. See `levels/levels_developer_guide.md`.

## Runtime Flow

1. `MainActivity` renders `GameApp`.
2. `GameApp` loads packaged level summaries from `assets/levels` through `levels/AssetLevelRepository`.
3. When the user starts a level, `createMatch` maps the authored level definition into `MatchState`.
4. While a level is running, the `LaunchedEffect` loop calls `stepMatch` once per rendered frame with a capped delta time.
5. `GameCanvas` renders the latest state, and tap handlers call `onScreenTap` or `upgradeBase`.

## Key Gameplay Rules

- Player sends launch `floor(current units * 0.5)` ships from a base.
- Fast bases launch fleets at higher movement speed.
- Bases produce units over time until they hit cap, and over-cap bases decay back toward cap.
- Base capacity is tiered by `capLevel`, with `cap = capLevel * 10`; each upgrade raises both by one tier until `maxLevel`, after which the selected-base upgrade pill is hidden. Upgrade prices use round tens based on the current level: `20` for level 1 to 2, `30` for level 2 to 3, `40` for level 3 to 4, and so on.
- The in-match upgrade affordance can still animate the most recent level-up even when that upgrade reaches `maxLevel`, because the animation follows the selected player base before the button visibility check removes the pill. The ring animation now fills over five seconds so the upgrade feedback reads as a longer-lived state change instead of a flash.
- Funds accrue for each non-neutral owner at a slower passive rate of `0.6 + 0.25 * owned bases` per second before campaign cash-rate multipliers apply, which keeps early cap upgrades from snowballing too quickly.
- Campaign upgrade levels currently cost `1` spendable star each and cap at level `5`. `Cash Flow` adds `+25%` mission income per level, `Refill Rate` adds `+25%` player-node ship production per level, and `Fleet Speed` adds `+20%` player fleet movement speed per level.
- The player can equip exactly one special ability at a time, each ability tracks its own upgrade level up to `5`, those ability levels persist even when the player switches the equipped loadout, and the equipped ability is snapshotted into the mission when the level starts or restarts.
- Special abilities are player-triggered from the in-match HUD. Timed buffs stay active for `5` seconds and then recharge for `30` seconds. The refill ability still uses the same recharge but applies its main top-up immediately and uses the five-second window as a follow-up production surge on the selected player node.
- Level completion awards between 1 and 3 stars based on authored completion-time thresholds stored in the level JSON; replaying a level improves the stored record only when the new run earns more stars, and only the improvement delta is added to spendable star reserves.
- Capturing a base reduces its `capLevel` by two tiers, but never below 1; surviving attackers keep their full post-combat count even if that temporarily exceeds the new captured cap.
- Friendly reinforcements also keep their full arrival count even if that temporarily pushes the destination base above its current cap; normal production then decays any over-cap garrison back toward cap over time.
- Tapping player-owned bases only changes selection state, empty-space taps clear the current selection, and tapping enemy or neutral targets with nothing selected shows only a brief onboarding hint instead of a persistent help banner; the message banner is reserved for upgrades, pause, win, loss, and error states rather than routine successful launches. Short-lived hints now remain visible until their expiry instead of being cleared early by a corrective base-selection tap, but a valid launch clears the hint immediately.
- The in-game HUD does not expose per-AI money totals, level title text, or a remaining-rivals chip; it keeps only a compact player-funds readout, the live mission timer, and the pause action on screen so the playfield stays less cluttered, with the timer separated from the funds row to reduce crowding on narrow devices.
- Android back follows the same screen flow as the visible menu actions: `LEVELS` and `UPGRADES` return to `HOME`, a running mission pauses first, and a second back from the paused overlay or a result overlay exits to the mission list.
- Each configured AI controller evaluates nearby non-owned bases every five seconds and attacks when it has a large enough unit advantage; otherwise it buys cap upgrades until a base is both full and at `maxLevel`, at which point that base pressures the weakest nearby non-owned target only when it has at least a modest numerical edge instead of idling or suiciding into a stronger defense.
- Route generation switches to a graph search across obstacle perimeter waypoints whenever a direct line intersects an obstacle buffer, so fleets follow a shortest visible polyline that stays outside the inflated obstacle clearance radius; waypoint generation respects named world safe margins, and if those constraints still leave no graph path, the helper falls back to the direct endpoint.

## Extension Points

- Add new authored entity fields in `levels/` before wiring them into runtime behavior.
- Add new base or fleet modifiers in `GameModel.kt` and apply them in `GameLogic.kt` plus `GameCanvas.kt`.
- Keep UI-only changes in `GameScreens.kt` or `GameCanvas.kt`; keep deterministic gameplay rules in `GameLogic.kt` so they stay unit-testable.
- When adding a new AI owner, AI type, node type, or authored level field, update the external editor in the same change so runtime and authoring stay on one schema.
